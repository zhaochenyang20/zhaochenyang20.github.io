基础命令

git status

  • On branch master 表明当前处于的分支，Git 默认为初始化的仓库创建 master 分支。
  • Untracked files 列出了还没有被 Git 管理的文件。

Git 不会主动管理目录下的文件，你得手动 add。

  • Changes to be committed 列出了此时进行 commit 操作将包括的变更，这里即添加了 README.md 文件。

Warning!!! 如果不小心添加了一个文件，或者一个更改还没准备好发布为 commit，可以遵循Git的提示执行 git rm --cached <file>，如果没有 --cached 参数则仓库中的 <file> 文件会被删掉，而且这几乎是不可逆的。（我室友就这么删掉了他大一所有代码）

这个指令真的很不安全，建议采用下文所述的 git restore --staged <file> 指令

git branch -a

另一分支的当前状态是自己的库的当前情况，而不是 commit 情况，故而另一分支也能看到 readme.md

git checkout -b dev

  • git 对于分支的保护作用，如果你切换到另一分支后，修改的工作不 commit，然后强行切换回来，那么你的修改会被 modified 掉，所以 git 会拒绝你切换回去，要求你在该分支上 commit 了，才能切回去

merge

merge A into B 的意义是，将 A 合入 B 中，不改变 A，但是改变 B

如果纯粹只用 merge A into B 的指令，会进入 vim 界面，然后编写理由，解释为什么这次 merge 是必要的，所以采用 merge A into B -m "anything" 的指令来 merge.

工作区与暂存区

学了这么久，这两个概念就没学清楚过。工作区就是 Working Tree，而使用了 git add <file> 之后，会把这个文件从工作区添加到暂存区 index/Stage/Cache，而git commit 是把暂存区提交到某个分支上的版本库上，也即 Repository/History。

git restore

git restore --staged

将暂存区的文件从暂存区撤出，但不会更改文件

git restore

  • 将不在暂存区（也即仅仅在工作区）的文件撤销修改，回到上一次 commit。

  • 不影响暂存区的修改，即便文件加入暂存区后再次被修改而没再加入暂存区。

  • 如果一个不在暂存区的文件在上次 commit 中尚且没有存在，也即并未被 git 追踪，那么对其 restore 将会失败。

  • 我们可以得出结论，暂存区和工作区只区分文件，而不区分文件内的修改。

git checkout <file>

git checkout -- file 也可以丢弃工作区的修改：

命令git checkout -- readme.txt意思就是，把readme.txt文件在工作区的修改全部撤销，这里有两种情况：

一种是readme.txt自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；

一种是readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。

git checkout -- file命令中的--很重要，没有--，就变成了“切换分支”。

总之，就是让这个文件回到最近一次git commit或git add时的状态。

对文件操作的总结

  1. git restore --staged <file> 用于无损地将文件撤出暂存区。
  2. git checkout -- <file> 用于将任何文件退回到上次 git add 或者 git commit 后的状态。
  3. 不要用 git rm --cached <file> 与 git restore <file>

版本回退

git reflog

这一指令显示每一步 commit 的移动关系，从上到下，每两行为一个单位。

每个单位的第一行是 destination，第二行是 departure。

比如第一个单位，从 d1e67d3 到 5ef2adf，对应着我使用了一次 > git reset --hard HEAD^

第二个单位，从 9e2be6a 到 5ef2adf，对应着我在 master 分支上的一次提交。注意，在上方轮廓图中，这两次提交中间还夹着 dev 分支的一次提交。实际上，9e2be 对应着我合并了 dev 到 master，然后又提交了一次 dev，产生了 5baba，之后再合并了一次，产生了 5ef2a。

git reset <version_id>

我个人不太喜欢用 HEAD^ 这种方式，感觉不详细。

有时候用 Git 的时候有可能 commit 提交代码后，发现这一次 commit 的内容是有错误的，那么有两种处理方法：

  1. 修改错误内容，再次 commit 一次。第一种方法比较直接，但会多次一次commit记录。

  2. 使用git reset 命令撤销这一次错误的 commit。我个人更倾向第二种方法，错误的commit没必要保留下来。

三种 reset 的模式

reset --hard：重置stage区和工作目录。

reset --hard 会在重置 HEAD 和 branch 的同时，重置 stage 区和工作目录里的内容。当你在 reset 后面加了 --hard 参数时，你的 stage 区和工作目录里的内容会被完全重置为和 HEAD 的新位置相同的内容。换句话说，就是你的没有 commit 的修改会被全部擦掉。而后你再 reset 回去也无法恢复。
