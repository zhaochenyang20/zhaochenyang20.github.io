结构化数据库

层级树状结构

XML、JSON 即是这种结构，但它无法很好地描述一些关系，如多对多。

网络数据结构

比树状的层级结构进步，但是仍然不能处理一些情况。

关系型数据库模型

用表来做直观展示，表有行和列，一行为一条记录，一列为一个字段，公共字段可以用于关联任何两张表。

结构化查询语言

SQL (Structured Query Language)，即结构化查询语言，则是对上述关系型数据库操作的利器。

SQL 也是一种编程语言，只是用途非常的特别。SQL本身有 ANSI 标准，不同数据库引擎会为自己的数据库而做出 一些语言上的调整，但是核心部分是一致的，这也是本课程的关注点。

SQL 在现今是直接或间接管理数据库必不可少的语言，常用于网站后端（非静态站点），而 SQL 写得好与坏也可以造成相同硬件配置下极大的性能差异。

关系数据库

上述模型，包括表的概念，甚至最初出现的 SQL，本身都属于理论，为了将它们实现，我们需要生成一个真正的关系数据库：一个（或一群）按照 SQL 语句可以高效完成对应操作的二进制文件

  3. 尽管语言规范上讲 SQL 的关键字（几乎）都是大写，但实际上常用数据库都是兼容大小写混用的，无论是类型还是下面语法中的关键字，因为懒得切输入法所以我可能难免会在演示中混用大小写

变量类型

SQL 作为一门编程语言，其中自然有若干变量类型，并且有对应的大小，如整型就有 INT（同 INTEGER）、SMALLINT、TINYINT、BIGINT 等，范围各不相同，但是常用的自然还是 INT。 常用的变量类型也正是我们在其他语言中经常使用的那些：

  • INT
  • FLOAT
  • CHAR
  • VARCHAR（相比 CHAR，存储时变长，节省空间，被使用的得更加广泛）
  • DATETIME

在写字符串常量时，我们会用单引号或双引号引起来，日期通常在交互时也会表现为字符串常量（本文不涉及日期）。

此外正如 Python 有 None，JavaScript 有 null，在 SQL 中也有个 NULL（空值）。

数据库实例

SQLite

SQLite 轻量但是强大，有一些特色语法。

SQLite 的一大特点是一个文件即一个数据库，默认情况下不涉及鉴权，这与它轻量易用的特点相符。（甚至不打开 文件也行，会直接在内存里建一个数据库）

SQLite 的系统管理语法均为定制的 . 打头的语法，如

  • .help：查看帮助
  • .q 或者 .quit：退出
  • .tables：显示当前数据库的所有表名
  • .schema ：查看表的 schema 信息（即如何创建）

此外也有 SQL 级的一些细微区别，如 SQLite 中不能使用 SELECT DISTINCT(*) FROM some_table;，但是 MySQL 可 以。

SQLite 的一大不足是它的事务控制不具备很好的并发性，理论上同时应当只有一个“用户”在使用它（唯一的例外是可以并行读），因此理论上不能在有并发性的网站中使用，否则可能造成数据库写入失败。

MySQL

MySQL 是一个商业级数据库，由于一些历史原因我选择使用开源的 MariaDB，这两者有极高的兼容性，不必太在意 tutorial 涉及到的 MySQL 与 MariaDB 差异。它有一套严格的权限管理（因此使用门槛也更高一些，需要额外的配置），可以并行访问，与上面的 SQLite 截然不同。

MySQL 有一系列关于鉴权的配置指令，这里不进行讲解，有兴趣可以自学。值得一提的是 information_schema 这张表，里面存储了大量配置信息，这属于 MySQL 的 feature。MySQL 有一些专用指令用于系统管理，如下是常用的一些指令：

  • show databeses;：显示自己有权限的数据库
  • use <database>：指明使用的数据库（或者连接时声明直接将数据库名写到命令行参数最后一项）
  • show tables;：显示当前数据库的所有表名
  • desc <table>;：给出当前数据库某张表的具体描述，类似于 SQLite 的 .schema <table>，但它给出的是描述而 非创建过程

如果真的要搭非静态网站，一般 MySQL / MariaDB 应该是首选。推荐使用 Docker 来降低配环境的成本。
